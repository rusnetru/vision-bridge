# vision-bridge — архитектура

Единый слой зрения для агента Mayya. Отдельный MCP-сервер, который прячет
разные способы «посмотреть на экран» за одним контрактом. Дата: 05.07.2026.

## 1. Цель и границы

**Цель.** Один интерфейс, чтобы Mayya (мозг на DeepSeek) могла:
- видеть экран десктоп-приложений и находить/отслеживать статичные элементы;
- работать с браузером так, чтобы сайт **не опознавал агента** и **не рвал сессию**.

**В scope:**
- Десктоп: обычные Windows-приложения (Win32/WPF/UWP) + Electron/Chromium.
- Браузер: два режима на выбор под задачу (реальный Chrome по CDP / отдельный стелс-браузер).

**Вне scope (сознательно выкинуто):** капча, игры, canvas, VLM-агенты (UI-TARS),
локальные vision-модели, GPU. Всё это в текущей задаче не нужно и не тянется.

## 2. Ключевое ограничение, которое определяет всё

Мозг Mayya — **текстовый** (`deepseek-chat`/`reasoner`). Он не принимает картинки.
Поэтому задача слоя — не «дать глаза», а строго:

> **экран → структурированный ТЕКСТ, который DeepSeek может прочитать и по которому может действовать.**

Отсюда: приоритет у методов, дающих текст/структуру (a11y-дерево, OCR-список
элементов), а не у методов, которые «сами смотрят на пиксели и сами кликают».

## 3. Контракт (сердце слоя)

Единая модель элемента. Каждый элемент несёт то, чем по нему кликать, — агенту
всё равно, браузер это, Блокнот или Slack.

```python
Element {
  id: str            # стабильная ссылка для агента, напр. "e17"
  role: str          # button | textbox | link | text | image | ...
  name: str          # видимый текст / подпись / accessible name
  value: str | None  # текущее значение (для полей)
  bbox: [x, y, w, h] # экранные координаты
  state: {enabled, focused, checked, visible, ...}
  backend: "uia" | "cdp" | "playwright" | "ocr"
  locator: Any       # то, чем ИМЕННО этот backend умеет кликать
}
```

Два вызова, которые видит агент:

```
capture(target, mode="auto") -> { elements: [Element], text: str, method_used, ts }
act(element_id, action, **args) -> { ok, error, new_state }
    action ∈ click | double_click | type | set_value | focus | scroll | read
find(query) / wait_for(query, timeout)  -> Element | None   # для статичных элементов
```

`target` — заголовок окна / «browser» / process. `text` — плоский текстовый
дамп экрана для случаев, когда агенту нужно просто «прочитать, что там».
Роутер выбора метода спрятан **внутри** `capture`, наружу не торчит.

## 4. Бэкенды

### 4.1 Десктоп
- **UIA (a11y) — основной.** Нативные Win32/WPF/UWP. Стабильные `AutomationId`/`Name`
  → надёжный `find`/`wait_for` по статичным элементам, чтение текста, клик, ввод.
  0 мс, без моделей, без GPU.
- **OCR fallback — для Electron/Chromium-окон**, где UIA частично слеп. Скриншот окна
  → Tesseract (текст) и/или OmniParser (кликабельные боксы, если нужны координаты
  элементов, а не только текст). CPU, без GPU. Включается роутером, только когда
  UIA вернул пусто/мусор.

Роутер десктопа: `UIA → (если бедно) OCR`.

### 4.2 Браузер — два режима (выбор под задачу)

Общий принцип незаметности разбит на две независимые проблемы:

**«Не рвать сессию»** решается не стелсом, а **постоянным реальным профилем**.
Одноразовый чистый Chromium (как в стандартном Playwright MCP) сайты видят как
новое устройство → переавторизация / разрыв. Работа в реальном профиле (те же
куки, тот же fingerprint, тот же IP) — рвать нечего.

**«Не опознать агента»** — прячем маркеры автоматизации: `navigator.webdriver`,
утечки CDP (`Runtime.enable`), бандл-Chromium без профиля.

Режимы:

- **Режим A — реальный Chrome по CDP (`connect_over_cdp`).** Chrome стартует с
  `--remote-debugging-port` и настоящим `--user-data-dir`. Слой подключается к уже
  живому браузеру. `navigator.webdriver=false`, настоящий отпечаток, живые куки —
  отличить от человека практически нельзя. Минус: браузер общий с пользователем.
  → для задач, где важна максимальная незаметность и сохранение логинов.
- **Режим B — отдельный стелс-браузер.** Свой Chrome под управлением слоя через
  **patchright** (пропатченный Playwright: прячет CDP-утечки) + копия профиля/кук
  пользователя + не-headless + человеческий темп действий. Агент и пользователь
  не мешают друг другу. Чуть выше теоретический шанс детекта, чем в режиме A.
  → для фоновой/параллельной работы.

Восприятие в обоих режимах — **a11y-снепшот страницы** (текстовое дерево с
`@ref`), как в текущем `browser-accessibility-first`. Скриншот — только по
запросу, DeepSeek его всё равно не читает.

> Честная оговорка: 100% необнаружимости не бывает (Cloudflare/DataDome
> развиваются). Связка «реальный профиль + patchright + человеческий темп»
> закрывает подавляющее большинство сайтов и полностью снимает разрыв сессии.

## 5. Подключение к Mayya

`vision-bridge` — отдельный процесс, **MCP-сервер** (FastMCP, Python). У Mayya уже
есть MCP-клиент (`src/mcp/client.py`, stdio) — подключается одной записью в
`mcp.json`. Отдельный процесс нужен, чтобы браузер и десктоп-зрение переживали
перезапуски агента и не тащили тяжёлые зависимости в код агента.

## 6. План этапов

**Этап 0 — контракт (0.5–1 день).** Зафиксировать `Element`/`capture`/`act`/`find`
/`wait_for`, поднять пустой FastMCP-сервер, подключить к Mayya, вернуть заглушки.
Это и есть «единый интерфейс» — фиксируем раньше всего.

**Этап 1 — десктоп через UIA (1–2 дня).** Обернуть существующий UIA за контракт:
`capture` окна → `[Element]`, `act` (click/type/focus), `find`/`wait_for` по
`AutomationId`/`Name`. Тест: Блокнот, Проводник, Telegram Desktop.

**Этап 2 — OCR fallback (1 день).** Роутер `UIA → OCR` для Electron. Скриншот окна
→ Tesseract/OmniParser → элементы. Тест: Slack / VS Code / Discord.

**Этап 3 — браузер (2–3 дня).** Режим A (CDP к реальному Chrome) + Режим B
(patchright + профиль). a11y-снепшот страницы за контракт, `act` по `@ref`.
Тест: залогиниться на сайте, уйти и вернуться — сессия жива; проверка на
`navigator.webdriver` и паре детект-тестов (bot.sannysoft, creepjs).

**Этап 4 — skill + тесты (1 день).** `skill vision-bridge` для Mayya с примерами,
10 сценариев (десктоп-нативный, Electron, браузер оба режима, поиск/ожидание
статичного элемента).

Итого ~5–7 дней. GPU не нужен ни на одном этапе.

## 7. Стек

- Python 3.11+, **FastMCP** (MCP-сервер).
- Десктоп: **uiautomation** / pywinauto (UIA), **pytesseract** (OCR), опц. OmniParser.
- Браузер: **patchright** (стелс-Playwright), CDP `connect_over_cdp`.
- Тесты: pytest; детект-проверки — bot.sannysoft.com, creepjs.

## 8. Риски

| Риск | Смягчение |
|---|---|
| Электрон-окна слепы для UIA | OCR-fallback (Этап 2) |
| Сайт всё же детектит | Режим A (реальный Chrome) + patchright; человеческий темп |
| Общий браузер мешает пользователю | Режим B (отдельный профиль) |
| OmniParser избыточен | Начать с Tesseract-текста; боксы — только если реально нужны координаты |
| Конфликт с текущим cua-driver/Playwright MCP | vision-bridge — отдельный MCP-сервер; старые отключать при тестах |
